# FridgeIR 实现文档：数据清洗 与 前端

> 面向工作展示。本文档详细说明 **数据清洗** 与 **前端** 两部分的设计目标、实现逻辑、关键算法、踩过的坑与最终结果。
> 项目定位：从食材出发的中文菜谱检索系统——用户输入手头食材 + 约束，返回"马上能做 / 差一两样就能做"的菜谱。

# 一、数据清洗

## 1.1 输入数据的真实情况

数据源：XiaChuFang Recipe Corpus（`recipe_corpus_full.json`）。第一步是"认清数据"，它和直觉差别很大：

| 特征 | 说明 | 对清洗的影响 |
|---|---|---|
| 伪装成 `.json` 的 **JSONL** | 每行一个 JSON 对象，不是一个大数组 | 必须按行解析，不能 `json.load` 整个文件 |
| 体量 **2GB / 155 万条** | — | 不能整体读进内存，必须**流式逐行**处理 |
| 字段 | `name / dish / description / recipeIngredient / recipeInstructions / author / keywords` | 食材/步骤字段名不是直觉的 `ingredients/steps` |
| **没有** 时间/难度/菜系 字段 | 语料里根本不存在 | 这些字段**不合成**，留给下游推断 |
| 食材"**用量在前**" | `"2大片生菜"`、`"半根黄瓜"`、`"1kg羊肉"`、`"适量盐"` | 解析方向与"名称在前"相反 |
| 各种脏数据 | HTML 实体 `可乐&#x9E21;翅`、全角字符、中文数字、分段标题 `【烫种】`、复合标注 `120克+-10克牛奶`、整组食材塞进一个数组元素、`dish="Unknown"`、`keywords` 多为 "XX的做法" SEO 垃圾 | 需要专门的归一与剥离逻辑 |

## 1.2 设计目标与原则

1. **只提取真实存在的字段，绝不凭空合成**（没有的时间/难度/菜系不造假）。
2. **流式处理**：内存占用与文件大小无关，能在普通笔记本上跑完 2GB。
3. **可配置**：原始字段名集中在 `FIELD_MAP`，换数据集只改映射，不改逻辑。
4. **可复现 + 可观测**：产出结构化数据 + 人工核对预览 + 质量报告。
5. **鲁棒**：单条坏数据跳过而非中断整批。

## 1.3 模块结构（`src/cleaning/`）

| 文件 | 职责 |
|---|---|
| `config.py` | 字段映射 `FIELD_MAP`、单位词典 `UNITS`、定性用量词、中文数字表、质量阈值 |
| `ingredient_parser.py` | 文本归一 + **从"用量在前"的食材文本中提取纯食材名** |
| `clean.py` | 流式读取 → 记录级提取 → 去重 → 写出 JSONL/CSV/报告（CLI 入口） |

运行：`python3 -m src.cleaning.clean`（在项目根目录）。

## 1.4 核心算法详解

### (1) 文本归一 `normalize_text`

所有字段进入处理前先统一"形":

```python
def normalize_text(s) -> str:
    s = html.unescape(str(s))            # &#x9E21; → 鸡（解码 HTML 实体）
    s = unicodedata.normalize("NFKC", s) # 全角→半角，兼容字符规整
    s = s.replace("　", " ")             # 全角空格
    s = re.sub(r"\s+", " ", s).strip()   # 合并空白
    return s
```

例：`可乐&#x9E21;翅` → `可乐鸡翅`；`２个` → `2个`。

### (2) 食材名提取 `extract_ingredient_name`（最难、最核心）

难点：中文菜谱食材是"**用量在前**"，且用量写法极其随意。更棘手的是——**食材名本身可能含中文数字**（"八角"含"八"、"三文鱼"含"三"、"五花肉"含"五"），早期版本按"第一个数字"切分会把 `五花肉` 切空、`八角` 丢失。

最终算法分四步（按顺序）：

```python
def extract_ingredient_name(raw):
    line = normalize_text(raw)
    if line[0] in "【「": return ""        # ① 烘焙分段标题（【烫种】）不是食材，丢弃
    line = _PAREN_RE.sub("", line)          # ② 去掉末尾括注 "(in spring water)"
    # ③ 循环剥离行首：定性词 / 约数词 / 数字(+大小中整修饰)+单位 / 连接标点
    prev = None
    while line and line != prev:
        prev = line
        for pat in (_LEAD_QUAL_RE, _LEAD_APPROX_RE, _LEAD_AMOUNT_RE):
            m = pat.match(line)
            if m: line = line[m.end():]
        line = _LEAD_PUNCT_RE.sub("", line)
    # ④ 兼容"名称在前"写法："西红柿 2个" → 去掉末尾用量
    m = _END_AMOUNT_RE.search(line)
    if m and m.start() > 0: line = line[:m.start()]
    return line.strip()
```

关键正则与设计意图：

| 正则 | 作用 | 设计要点 |
|---|---|---|
| `_LEAD_AMOUNT_RE` | 行首"数字(+大/小/中/整)+单位" | 数字字符集含中文数字；单位后置，**只在行首剥离**，避免误伤名称里的数字 |
| `_LEAD_QUAL_RE` | 行首定性词（适量/少许…） | `"适量盐"` → `盐` |
| `_END_AMOUNT_RE` | 行尾用量 | 兼容"名称在前"，且**锚定行尾**，所以 `五花肉500克` 的 `五` 不会被当用量 |
| 循环 `while` | 反复剥离 | 应对复合写法 `120克+-10克牛奶`：剥 `120克` → 剥 `+-` → 剥 `10克` → `牛奶` |
| `UNITS` 按长度降序 | 单位匹配 | 防止 `千克` 被 `克` 截断 |

实测效果（迭代中真实修复的 case）：

| 原始文本 | 提取结果 | 说明 |
|---|---|---|
| `2大片生菜` | `生菜` | 用量在前 + 大小修饰 |
| `半根黄瓜` | `黄瓜` | 中文数字 `半` |
| `1kg羊肉` | `羊肉` | 英文单位 |
| `适量盐` | `盐` | 定性用量 |
| `2个八角` | `八角` | **名称含数字字符，不能误切** |
| `120克+-10克牛奶` | `牛奶` | 循环剥离复合标注 |
| `【烫种】` | `（丢弃）` | 分段标题非食材 |
| `可乐&#x9E21;翅` | `可乐鸡翅`（菜名场景） | HTML 实体解码 |

### (3) 食材字段解析 `extract_ingredient_names`

原始 `recipeIngredient` 可能是 `list[str]`、`list[dict]` 或一整段多行字符串，统一归一：

```python
if isinstance(raw, str): items = re.split(r"[\n;；,，、]+", raw)   # 多行/分隔符拆分
...
out = [(原文, 食材名) for it in items if 食材名 != ""]
```

保留 `(原文, 食材名)` 二元组——**原文用于展示，食材名用于检索匹配**，两者都不丢。

### (4) 步骤清洗 `clean_steps`

```python
s = normalize_text(s)
s = re.sub(r"<[^>]+>", "", s)                         # 去 HTML 标签
s = re.sub(r"^\s*\d+\s*[\.、,，:：)）]\s*", "", s)     # 去"1. / 2、 / 3）"序号前缀
```

### (5) 记录级提取 `extract_record` + 质量门槛

```python
name 为空            → 丢弃 (empty_name)
食材数 < 2 (MIN_INGREDIENTS) → 丢弃 (few_ingredients)
步骤数 < 1 (MIN_STEPS)       → 丢弃 (few_steps)
dish == "Unknown"           → 置空
```

### (6) 流水线 `run`：流式 + 去重 + 产物

- **流式**：`for line in f` 逐行，内存恒定（2GB 文件也只占常量内存）。
- **去重**：以 `(菜名, 排序后的食材集合)` 的 `hash` 为签名，落进 `set`，重复则丢弃。
- **自增 ID**：`r0000001` 形式，稳定、可被下游（后端 `source_recipe_id`）引用。
- **三份产物**：
  - `recipes.jsonl`：结构化主数据（下游建索引/导库用）
  - `recipes_preview.csv`：前 300 条，给人肉眼核对
  - `quality_report.json`：各项计数 + 平均食材数

## 1.5 输出 Schema（`recipes.jsonl` 每行）

```json
{
  "id": "r0000067",
  "name": "超美味的西红柿蛋汤",
  "dish": "西红柿蛋汤",
  "description": "...",
  "ingredients": [{"raw": "2个西红柿", "name": "西红柿"}, ...],
  "ingredient_names": ["西红柿", "鸡蛋", "盐", "淀粉"],

  "steps": ["热油下葱花爆锅", ...],

```

## 1.6 最终结果

| 指标 | 数值 |
|---|---|
| 原始 | 1,550,151 条 |
| **清洗后保留** | **1,486,919 条（96%）** |
| 丢弃-食材<2 | 44,190 |
| 丢弃-步骤<1 | 117 |
| 丢弃-重复 | 18,925 |
| 平均食材数 | 7.32 |
| 耗时 | ≈ 86 秒（单进程，普通笔记本） |

## 1.7 数据子集抽取（`src/retrieval/subset.py`，数据准备）

为便于开发/演示/评估，从全量抽 **2500 条"种类丰富"** 的子集：

- **主食材分桶 + 轮转抽样**：以每条菜的"主食材"分桶，轮转各取一条 → 2500 条覆盖 **2500 种不同主食材**，不堆同类。
- **质量过滤**：剔除食材名没切干净（含 `克/：/主料`）、步骤过少、食材组合重复的记录。
- **两遍扫描省内存**：第一遍只读元数据选行号，第二遍按行号取整条。

## 1.8 设计权衡与已知局限

- 少数菜谱把"一整组食材"塞进 `recipeIngredient` 的单个元素（用 `、丶` 分隔未切开），残留脏食材名——保留了 `raw` 原文，下游子串匹配可部分补偿。
- 时间/难度/菜系**不在清洗阶段合成**：原始数据没有，硬造会污染数据，交给后端服务在导入时按规则推断（保证清洗产物"干净可信"）。

# 二、前端

## 2.1 技术选型

| 选择 | 理由 |
|---|---|
| **Vite + React + TS** | 后端接口文档直接给了 TS 类型，前后端契约可在编译期校验；Vite HMR 快、构建产物小（gzip ≈ 50KB） |
| **Vite 代理** `/api → 127.0.0.1:8000` | 前端用同源相对路径请求，**彻底绕开 CORS**，无需后端配置跨域 |
| **纯 React useState**（不引状态库） | 单页应用、状态量可控，避免过度工程 |

## 2.2 目录结构与分层

```
frontend/src/
├── types.ts        ── 与后端接口一一对应的 TS 契约（唯一事实源）
├── api.ts          ── 数据层：封装接口 + 统一错误处理 + 健康检查
├── bucket.ts       ── 工具：bucket 文案 → 颜色/图标
├── App.tsx         ── 状态编排 + 页面骨架
├── components/
│   ├── TagInput.tsx         ── 食材 chip 输入框
│   ├── SegmentedControl.tsx ── 泛型分段选择器
│   ├── Filters.tsx          ── 偏好筛选面板
│   ├── ResultCard.tsx       ── 搜索结果卡片
│   └── RecipeDetailModal.tsx── 详情弹窗 + AI 改良
└── styles.css
```

分层原则：**类型(契约) → 数据(api) → 展示(components) → 编排(App)**，单向依赖，组件不直接 fetch。

## 2.3 前后端接口契约

前端覆盖后端全部面向用户的能力：

| 后端接口 | 前端用途 |
|---|---|
| `POST /api/v1/search/by-ingredients` | 主搜索（食材 + 排除 + 偏好 + 分页） |
| `GET /api/v1/recipes/{id}` | 菜谱详情 |
| `POST /api/v1/recipes/{id}/enhance` | AI 改良做法（DeepSeek） |
| `GET /health` | 后端连接状态指示 |

后端字段 → 前端展示映射（节选）：

| 后端字段 | 前端展示 |
|---|---|
| `parsed.ingredients[].canonical` | 顶部"识别到"tag（`西红柿→番茄`） |
| `parsed.need_confirmation` | **低置信度食材确认提示框**（见 2.8） |
| `items[].bucket` | 卡片彩色徽章（✅马上能做/🛒再买1样/🧺还差几样/💡灵感参考） |
| `items[].matched / missing` | 绿色"已匹配" / 橙色"还缺" chip |
| `items[].recipe_tags / preference_matches` | 标签，命中偏好的高亮 |
| `items[].reason / rerank_reason` | 推荐原因 / AI 重排说明 |
| `ingredients[].required` | 详情页拆分"必需食材"与"基础调味品" |

## 2.4 数据层 `api.ts`

统一 `request()` 封装，关键是**错误透传**：

```ts
if (!res.ok) {
  let detail = "";
  try { detail = (await res.json())?.detail } catch {}   // FastAPI 错误体 {detail}
  if (res.status === 404) throw new Error(detail || "菜谱不存在");
  if (res.status === 422) throw new Error(detail || "搜索参数错误");
  if (res.status >= 500) throw new Error(detail || "服务异常，请稍后重试");
}
```

效果：AI 改良未配 key 时，后端返回 `503 {detail:"未配置 DEEPSEEK_API_KEY"}`，前端**原样显示后端原因**而非笼统报错。

## 2.5 状态编排 `App.tsx`

核心状态：搜索输入（`items / excluded / filters / page`）、结果（`result / loading / error`）、详情（`detail / selected`）、后端状态（`backendOk`）、确认框（`confirmDismissed`）。

一个关键工程细节——**闭包陈旧问题**：

```ts
const doSearch = async (toPage = 1, itemsArg: string[] = items) => { ... };
const tryExample = (list) => { setItems(list); doSearch(1, list); };  // 传入覆盖值
```

`setItems` 是异步的，"点击示例食材立即搜索"时若直接读 `items` 会拿到旧值，故 `doSearch` 支持 `itemsArg` 覆盖参数。

## 2.6 组件详解

| 组件 | 实现要点 |
|---|---|
| **TagInput** | 回车 / 逗号 / 顿号分词成 chip；空输入按 Backspace 删最后一个；失焦自动提交；去重 |
| **SegmentedControl** | TS 泛型 `<T extends string｜boolean｜null>`，"不限/A/B"三态分段选择 |
| **Filters** | 把辣度/荤素/难度/分量/调料映射到 `filters` 字段；烹饪手法多选（炒蒸煎拌炖**炸**）；两个开关 |
| **ResultCard** | bucket 徽章（图标+语义色）、matched/missing chip、标签（命中偏好高亮）、reason、点击进详情 |
| **RecipeDetailModal** | 详情 + 必需食材/调味品分组 + 步骤；内置 **AI 改良**：调 `enhance`，独立 loading/error，切换菜谱时按 `recipe_id` 重置 |
| **bucket.ts** | `缺0→马上能做(绿✅) / 缺1→再买1样(蓝🛒) / 缺2-3→还差几样(橙🧺) / 缺4+→灵感参考(灰💡)` |

## 2.7 关键交互流程

```
输入食材+偏好 ─click─▶ doSearch ─POST search─▶ 渲染 解析tag + 确认提示 + 结果卡片 + facets + 分页
                                                        │ click 卡片
                                                        ▼
                                            openDetail ─GET recipe/{id}─▶ 详情弹窗
                                                        │ click「AI 改良做法」
                                                        ▼
                                            enhance ─POST enhance─▶ 生成做法(标题/食材/步骤/贴士)
```

## 2.8 UX 优化点（展示亮点）

- **空状态引导**：首屏不是空白，而是 🍳 大图标 + "冰箱里有啥" + **示例食材一键试**（点"西红柿+鸡蛋"自动填入并搜索）。
- **bucket 彩色徽章**：把抽象的"缺几样"翻译成一眼可懂的标签 + 颜色 + 图标。
- **低置信度确认提示框**：搜索后读取 `parsed.need_confirmation`，若后端对某些食材识别置信度较低（如"鱼""萝卜"），在结果区顶部弹出醒目黄色提示框「⚠️ 以下食材识别置信度较低，请确认是否正确：鱼 萝卜」，可点 × 关闭，每次搜索自动重置。采用"搜索后统一提示"方案——只调搜索接口、读 `data.parsed.need_confirmation`，无需单独调用 `/ingredients/parse`，简化前端。
- **后端连接状态**：右上角实时显示"● 后端已连接/未连接"，断连时给明确提示而非搜索失败。
- **错误友好化**：所有错误透传后端原因，区分 404/422/503。
- **响应式**：宽屏两栏（筛选侧栏 + 结果区），窄屏单栏且侧栏取消 sticky 避免遮挡。
- **AI 结果视觉区分**：改良做法用紫蓝渐变卡片，与原始菜谱区分，并带模型名与免责声明。

## 2.9 边界与容错

后端未连接 / 搜索空结果 / 低置信度食材（`need_confirmation` 确认框）/ AI 未配 key —— 均有对应 UI 文案，不会白屏或崩溃。
